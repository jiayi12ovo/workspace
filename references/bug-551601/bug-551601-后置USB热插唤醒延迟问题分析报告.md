# bug-551601 后置 USB 热插唤醒延迟问题分析报告

## 一、问题概述

联想开天 M90h G1t 在后置 USB 口开机无设备接入的情况下，系统静置后插入 U 盘，不能立即识别，通常需等待 30-60s 后才出现 USB 枚举和挂载。

该问题在 `ukui-power 3.26.0.0-0k11.4` 上复现。回退到旧版电源管理包后，后置 USB 插入 U 盘可立即识别。

## 二、影响环境

- 机型：联想开天 M90h G1t
- 主板：KHG3X00MB1
- BIOS：W0JKT22A
- EC：W0JCT0FA
- CPU：Hygon 3350
- OS：Kylin-Desktop-V10-SP1-2503-Release-OEM-LX
- 内核：5.4.18-142-generic
- 问题电源管理版本：3.26.0.0-0k11.4
- 对比正常版本：3.26.0.0-0k3.9oemlxx7h3.1.u.oemlx0.3.u

## 三、复现步骤

1. 后置 USB 口不接入任何 USB 设备。
2. 开机进入系统。
3. 系统静置，等待电源管理策略生效。
4. 在后置 USB 口插入 U 盘。

实际结果：U 盘需等待 30-60s 后才识别。  
期望结果：插入后立即识别。

## 四、现场现象

问题机器存在多个 USB 控制器：

- `06:00.3` Chengdu Haiguang USB 3.0 Host controller
- `03:00.0` Device `1f39:2300`

现场分析中发现，前置 `06:00.3` xHCI 控制器未进入 suspend，后置 `03:00.0` 控制器进入 suspend。

前置控制器未进入 suspend 是正常现象：其下游存在 USB 音频设备保持 active，电源管理组件配置 `auto` 时会屏蔽音频设备，因此该设备不休眠，上游 USB 总线和控制器也保持 active。

后置控制器在空口静置后进入 suspend，PCI 设备进一步进入 D3cold。此时插入 U 盘，正常情况下平台应上报 GPE/PME 唤醒事件，内核收到后唤醒 PCI/xHCI 控制器，再进行 root hub 恢复和 USB 枚举。

但该机器在此场景下无法收到 GPE 唤醒事件，xHCI 控制器不能被热插动作及时唤醒，因此 U 盘不能立即识别。

30-60s 后能够识别，是因为系统中的电源管理组件后续调整电源策略，手动唤醒了控制器。控制器恢复后，内核重新扫描总线，才发现插入的 U 盘。

## 五、根因判断

该问题不是 U 盘枚举慢，也不是 usb-storage 驱动问题，而是后置 USB 所属 xHCI 控制器进入低功耗后，空口热插唤醒链路失效。

触发条件包括两部分：

1. `ukui-power-management-service` 将 USB xHCI PCI 控制器的 `power/control` 设置为 `auto`。
2. 后置 USB 所属 xHCI 控制器进入 runtime suspend/D3cold 后，平台无法通过插入设备事件可靠唤醒控制器。

截图 `ukui-power.jpg` 中可见 `ukui-power-management-service` 批量写入 PCI 设备的 runtime PM 策略，例如：

```text
"/sys/bus/pci/devices/0000:03:00.0/power/control": "auto"
```

因此，直接触发原因是电源管理策略将不应进入 runtime suspend 的 USB 控制器设置为 `auto`；底层平台问题是该控制器进入 D3cold/runtime suspend 后，插入 USB 设备无法可靠触发 GPE/PME 唤醒。

## 六、内核默认策略说明

内核默认策略本身是保护性的，并不会默认放开所有 xHCI 控制器的 runtime PM。

在 `drivers/usb/host/xhci-pci.c` 中，只有 xHCI HCI version 大于等于 1.2 时，内核才设置 `XHCI_DEFAULT_PM_RUNTIME_ALLOW`：

```c
/* xHC spec requires PCI devices to support D3hot and D3cold */
if (xhci->hci_version >= 0x120)
	xhci->quirks |= XHCI_DEFAULT_PM_RUNTIME_ALLOW;
```

随后才会执行：

```c
if (xhci->quirks & XHCI_DEFAULT_PM_RUNTIME_ALLOW)
	pm_runtime_allow(&dev->dev);
```

也就是说，xHCI 1.1 控制器默认不会被内核主动设置为 runtime PM allow。当前问题是 `ukui-power` 越过了内核默认保护策略，主动写入 `power/control=auto`，从而暴露了该控制器低功耗唤醒异常。

## 七、责任边界

该问题可分为两层：

- 电源管理侧：旧版 `ukui-power` 策略过于激进，错误地将 USB controller 设置为 `power/control=auto`。
- 平台/固件/控制器侧：后置 xHCI 控制器进入 D3cold/runtime suspend 后，插入设备无法可靠触发 GPE/PME 唤醒。

内核默认策略实际规避了该风险，问题版本电源管理包改变了控制器 runtime PM 策略后暴露平台唤醒缺陷。

## 八、新版 ukui-power 处理情况

新版 `ukui-power` 包已经根据 `modias/modalias` 对 USB controller 进行屏蔽，不再对 USB controller 设置 `power/control=auto`。

该处理方向是正确的：对于本问题机器，应保持受影响 USB xHCI 控制器的 runtime PM 策略为 `on`，避免其在空口静置后进入 D3cold/runtime suspend。

建议新版验证点：

```bash
cat /sys/bus/pci/devices/0000:03:00.0/power/control
```

期望结果：

```text
on
```

同时确认 `ukui-power-management-service` 日志中不再对 USB controller 写入 `auto`。

## 九、建议修复方案

短期修复：

在 `ukui-power` 中继续保留 USB controller 黑名单逻辑，基于 `modias/modalias` 或 PCI class 识别 USB controller，不对其设置 `power/control=auto`。

建议保持：

```bash
echo on > /sys/bus/pci/devices/0000:03:00.0/power/control
```

长期修复：

联想/BIOS/平台侧需确认该 xHCI 控制器在 D3cold/runtime suspend 下的 GPE/PME 唤醒能力，修复空口插入设备无法唤醒控制器的问题。

## 十、验证标准

1. 升级新版 `ukui-power` 后，USB controller 不再被设置为 `power/control=auto`。
2. 后置 USB 空口开机并静置后，插入 U 盘可立即识别。
3. 问题控制器保持 `power/control=on` 时，无法复现 30-60s 延迟识别。
4. 手动设置问题控制器 `power/control=auto` 并等待 `runtime_status=suspended` 后，插入 U 盘仍可复现延迟，证明问题与控制器 runtime suspend 唤醒链路相关。
5. 延迟期间 dmesg 无 USB new device 枚举日志；控制器被唤醒后才出现枚举日志。

## 十一、结论

bug-551601 的本质是：旧版 `ukui-power` 错误将后置 USB 所属 xHCI 控制器设置为 `power/control=auto`，导致控制器进入 runtime suspend/D3cold；该控制器或平台固件在低功耗状态下无法通过 USB 热插事件可靠上报 GPE/PME 唤醒，最终表现为插入 U 盘后需等待 30-60s 才能识别。

新版 `ukui-power` 已根据 `modias/modalias` 屏蔽 USB controller，避免继续触发该问题。后续若需彻底解决低功耗唤醒能力问题，需要平台/BIOS/控制器侧进一步修复。
