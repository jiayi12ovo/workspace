# bug-569193：S4 休眠镜像与缓存关系分析

## 背景

当前问题是：在该平台上，卸载 WPS 后系统缓存明显减少，但 S4 休眠镜像大小和唤醒耗时没有同比例下降。

排查目标是：确认 S4 休眠镜像里到底保存了哪些内容，找出减少 S4 唤醒时间的有效方向。

当前目录中的关键材料：

- `havewps.txt`：卸载 WPS 前，运行 `analyze_cache.sh` 的输出。
- `havewps_comlog_57s.txt`：卸载 WPS 前的 S4 日志。
- `dlewps.txt`：卸载 WPS 后，运行 `analyze_cache.sh` 的输出。
- `delwps_com_log_55s.txt`：卸载 WPS 后的 S4 日志。
- `free_mem.txt`：卸载 WPS 后执行 `echo 3 > /proc/sys/vm/drop_caches`，再运行 `analyze_cache.sh` 的输出。
- `free_mem_comlog.txt`：卸载 WPS 后 drop cache 场景的 S4 日志。
- `bug-569193-hibernate-image-trace.patch`：用于统计休眠镜像页类型的内核调试补丁。

## 当前现象

这不是简单的“cache 不会进入休眠镜像”的问题。

从数据看，cache 确实会影响 S4 镜像：

- drop cache 后，`buff/cache` 从 2620 MB 降到 715 MB。
- S4 镜像从约 4.4 GB 降到约 3.1 GB。
- 镜像读取耗时从 18.52 s 降到 12.12 s。

但异常点在第一段：

- 卸载 WPS 后，`buff/cache` 从 4773 MB 降到 2620 MB，减少约 2.1 GB。
- 但 S4 镜像只从约 4.9 GB 降到约 4.4 GB，只减少约 0.5 GB。

也就是说：卸载 WPS 减少的那部分缓存，和后续 `drop_caches` 释放掉的缓存，对休眠镜像的贡献比例不同。WPS 相关缓存在该平台上没有按 1:1 进入休眠镜像；而卸载 WPS 后系统里剩下的缓存，反而更大比例体现在休眠镜像里。

## 为什么 `free -h used=1.6G` 不能直接对应 3.1G 镜像

`free -h` 里的 `used` 不是休眠镜像的统计口径。

以 drop cache 场景为例：

- `MemTotal`: 15862744 KB
- `MemFree`: 13433300 KB
- `MemTotal - MemFree`: 约 2.3 GB
- `free -h used`: 约 1.6 GB
- `buff/cache`: 715 MB
- S4 镜像读取大小：约 3.1 GB

`free -h used` 会把很多 `buff/cache` 从 used 里扣掉，但 S4 snapshot 关心的是哪些物理页需要保存和恢复。休眠镜像里可能包含：

- 用户进程匿名页，例如堆、栈、匿名 mmap；
- tmpfs/shmem；
- 被进程映射的文件页，例如共享库、桌面组件、驱动用户态库；
- 仍然保留的 clean/dirty file cache；
- slab 和其他内核内存；
- 页表；
- 驱动分配的内存；
- 休眠镜像元数据中记录的 zero pages。

因此，`used=1.6G` 和 S4 镜像 3.1G 不一致，并不代表数据矛盾。更接近的运行时口径是 `MemTotal - MemFree`，drop cache 场景下约为 2.3G，但它仍然不能完全等价于休眠镜像大小。

## 当前证据缺口

目前 bug-569193 的日志主要能看到恢复阶段的镜像加载：

```text
PM: Loading and decompressing image data (... pages)
PM: Read ... kbytes in ... seconds
```

但缺少创建休眠镜像阶段的关键日志：

```text
PM: Creating hibernation image:
PM: Need to copy ... pages
PM: Hibernation image created (... pages copied, ... zero pages)
```

更关键的是，目前还不知道这些被保存的页分别属于什么类型，例如匿名页、文件页、mapped 文件页、slab、页表或驱动内存。因此，仅靠 `free -h`、`buff/cache` 和恢复阶段读取大小，无法精确回答“镜像里到底是什么”。

## 已准备的内核调试补丁

补丁路径：

```text
references/bug-569193/bug-569193-hibernate-image-trace.patch
```

补丁功能：

- 增加 `/sys/power/hibernate_image_trace` 动态开关。
- 增加 `/sys/power/hibernate_image_trace_stats` 结果读取节点。
- 在创建休眠镜像时，在 `copy_data_pages()` 中按实际进入镜像的源 PFN 统计页类型。
- 在 `PM: Hibernation image created` 后打印分类统计。

使用方式：

```bash
echo 1 | sudo tee /sys/power/hibernate_image_trace
systemctl hibernate
```

唤醒后查看：

```bash
dmesg | grep -A20 "Hibernate image page breakdown"
cat /sys/power/hibernate_image_trace_stats
```

预期统计项：

- `total`：进入镜像统计范围的总页数；
- `copied`：实际拷贝的非零页；
- `zero`：全零页；
- `anon`：匿名页；
- `shmem`：tmpfs/shmem 页；
- `mapped_file`：被进程映射的文件页；
- `file_clean`：未映射的干净文件页；
- `file_dirty`：未映射的脏文件页；
- `slab`：slab 页；
- `pagetable`：页表页；
- `reserved`：reserved 页；
- `kernel_other`：其他内核或驱动相关页；
- `unknown`：未分类页。

## 建议补充测试

建议在打入 trace patch 后补齐以下测试矩阵：

| 测试项 | 目的 |
|---|---|
| 有 WPS，不 drop cache | 建立 WPS 存在时的基线 |
| 卸载 WPS，不 drop cache | 复现当前异常场景 |
| 卸载 WPS，drop cache | 复现当前 3.1G 镜像场景 |
| 有 WPS，drop cache | 缺失的关键对照组 |

每组休眠前采集：

```bash
cat /sys/power/image_size
cat /sys/power/reserved_size
egrep 'MemFree|MemAvailable|Buffers|Cached|SReclaimable|AnonPages|Mapped|Shmem|Slab|SUnreclaim|KernelStack|PageTables|Percpu|Active\(file\)|Inactive\(file\)|Dirty|Writeback|Unevictable' /proc/meminfo
cat /sys/power/hibernate_image_trace
systemctl hibernate
```

唤醒后采集：

```bash
dmesg | grep -E 'Need to copy|Hibernation image created|Hibernate image page breakdown|Read [0-9]+ kbytes' -A20
cat /sys/power/hibernate_image_trace_stats
```

## 当前判断

当前比较明确的判断是：

1. 卸载 WPS 后，系统可见缓存确实减少约 2.1 GB，但这部分只带来约 0.5 GB 的 S4 镜像下降。

2. 卸载 WPS 后再执行 `drop_caches`，缓存继续减少约 1.9 GB，同时 S4 镜像下降约 1.3 GB，说明剩余 cache 中有相当部分会进入休眠镜像。

3. drop cache 后仍有约 3.1 GB 休眠镜像，说明当前平台/桌面会话存在一个较高的基础镜像大小，不能只从 WPS 文件缓存解释。

4. 3.1 GB 基础镜像的大头可能来自匿名页、mapped 文件页、内核不可回收内存、页表或驱动分配。cx4 显卡栈值得重点关注，因为日志中存在较多 cx4 初始化、shared VM、page table 相关信息。

5. 下一步必须依赖创建休眠镜像时的页类型统计，才能确认优化方向。如果 `file_clean/mapped_file` 大，继续查文件缓存和映射页；如果 `anon` 大，查常驻进程；如果 `kernel_other/slab/pagetable` 大，结合 `page_owner` 查驱动和内核分配。

## 结论

当前现象可以概括为：

```text
卸载 WPS：缓存少约 2.1 GB，但 S4 镜像只少约 0.5 GB。
drop cache：缓存再少约 1.9 GB，S4 镜像少约 1.3 GB。
```

因此，减少 S4 唤醒时间不能只看 `free -h` 或 `buff/cache` 总量，需要确认创建休眠镜像时实际保存的页类型。当前已经准备了内核 trace 补丁，用于直接统计 S4 镜像中的页类别。
